# Gigabrain v0.5.1

`v0.5.1` makes Gigabrain a first-class Codex App and Codex CLI memory stack without depending on undocumented Codex internals.

## What shipped

- Codex standalone setup is now public and npm-installable via `gigabrain-codex-setup`
- Gigabrain now exposes a stable MCP surface for Codex:
  - `gigabrain_recall`
  - `gigabrain_remember`
  - `gigabrain_checkpoint`
  - `gigabrain_provenance`
  - `gigabrain_recent`
  - `gigabrain_doctor`
- Codex App gets a dedicated checkpoint lane for episodic session capture into `~/.codex/gigabrain/memory/YYYY-MM-DD.md` by default
- Setup now generates `.codex/setup.sh` plus helper actions for MCP install, verification, maintenance, and manual checkpointing
- Codex setup now initializes both the shared repo store and the shared personal user store, and it migrates older standalone configs onto the new defaults

## Codex memory model

`v0.5.1` keeps the memory split explicit:

- `~/.codex/gigabrain/MEMORY.md` is the shared repo-store memory layer by default
- `~/.codex/gigabrain/profile/MEMORY.md` is the shared personal user-memory layer by default
- `~/.codex/gigabrain/memory/YYYY-MM-DD.md` stores task-end session checkpoints and other episodic native notes
- `~/.codex/gigabrain/memory/registry.sqlite` remains the operational recall/query layer
- repo-scoped continuity defaults to the derived `project:<repo>:<hash>` scope, so different repos stay separated even when they share the same top-level store

Codex targeting in `v0.5.1` is now explicit:

- use `target=user` for stable personal preferences and facts that should follow you across repos
- use `target=project` for repo-specific decisions, conventions, and work-in-progress context
- `gigabrain_checkpoint` stays repo-scoped by default and writes into the repo store's native daily log

This release does **not** add hidden transcript logging to Codex App. Instead:

- explicit durable saves still go through `gigabrain_remember`
- meaningful task-end summaries go through `gigabrain_checkpoint`
- consolidation is manual in App mode via `maintain`

That keeps Codex support honest, local-first, and compatible with documented Codex integration points.

## Why the checkpoint lane matters

OpenClaw/Nimbus has hook-driven capture paths that Codex App does not expose publicly. `v0.5.1` solves that gap with intentional task-end session checkpoints inside the shared Codex store:

- after a completed implementation
- after a debugging pass
- after a planning/compaction-style summary
- before closing a session with decisions or open loops

These checkpoints are native-only at write time, then later become available for:

- native recall immediately after sync
- later promotion into durable registry memories when the content proves valuable

## Install paths

OpenClaw and Codex now have clearly separate setup paths:

- OpenClaw plugin path: `npm run setup -- --workspace /path/to/workspace`
- Codex standalone path:
  - `npm install @legendaryvibecoder/gigabrain`
  - `npx gigabrain-codex-setup --project-root /path/to/repo`

The default Codex standalone install now uses one personal store across repos. If someone wants strict isolation instead, they can opt into `--store-mode project-local`.

After Codex setup, register the MCP server with the generated `codex mcp add gigabrain ...` command.

Recommended Codex verify path:

- `npx gigabrainctl doctor --config ~/.codex/gigabrain/config.json --target both`
- `.codex/actions/verify-gigabrain.sh`
- `gigabrain_doctor target=both` from inside Codex

Upgrading a pre-fix Codex install:

- re-run `npx gigabrain-codex-setup --project-root /path/to/repo`
- this migrates empty `codex.userProfilePath`, restores the repo-derived default project scope, and updates recall order to `project,user,remote`

## Validation

Release validation for `v0.5.1` covered:

- full automated test suite
- fresh Codex setup integration
- Codex setup migration from the earlier empty-user-store configuration
- MCP startup and tool registration
- MCP user-store writes plus health checks for both repo and user stores
- checkpoint write + immediate recall from native logs
- package tarball dry run

## Operational notes

- Codex App support is MCP-first and local-first
- there is still no background daemon/nightly loop for Codex App
- manual `maintain` is the intended consolidation path in standalone Codex mode
- explicit `target=user` health checks now fail honestly when the personal store is not configured yet
- OpenClaw plugin behavior remains intact; `v0.5.1` adds Codex support without breaking the existing plugin path
