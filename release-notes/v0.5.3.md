# Gigabrain v0.5.3

Status: stable.

`v0.5.3` is a release-quality hardening pass for Gigabrain as a local-first memory layer across OpenClaw, Codex, Claude Code, and Claude Desktop. It brings Claude into the supported standalone path, fixes OpenClaw fresh-user onboarding, tightens recall behavior from PR `#33`, and adds stronger release validation so the package is safer for new users.

## Top improvements

- Claude Code now has a first-class setup flow via `gigabrain-claude-setup`
- Claude Desktop now gets both a local-test `.dxt` bundle flow and a portable release-bundle flow
- OpenClaw onboarding now follows the current `openclaw plugins install` / extensions model
- Recall routing is more resilient on identity, preference, temporal, and noisy metadata-heavy prompts
- Release validation now covers packaged installs, live Codex/OpenClaw host smokes, a deep recall eval lane, and portable Claude bundle behavior

## Install / upgrade notes

- **OpenClaw**: use `openclaw plugins install @legendaryvibecoder/gigabrain`, then rerun `npm run setup -- --workspace ...`, then run `npx gigabrainctl doctor --config ~/.openclaw/openclaw.json`
- **Codex**: rerun `npx gigabrain-codex-setup --project-root /path/to/repo` to refresh the shared standalone config and helper scripts. Fresh installs now default to `~/.gigabrain/config.json`; existing `~/.codex/gigabrain/config.json` installs remain supported in place.
- **Claude Code / Claude Desktop**: run `npx gigabrain-claude-setup --project-root /path/to/repo`, then `.claude/actions/verify-gigabrain.sh`, then build/import the desktop `.dxt` if needed

## Claude support

### Claude Code

Claude Code now gets a proper setup flow instead of requiring manual MCP wiring.

`gigabrain-claude-setup` bootstraps the shared standalone store, preserves the same repo-scoped project memory and shared personal user memory model used by Codex, and writes the Claude-facing artifacts:

- `CLAUDE.md`
- `.mcp.json`
- `.claude/setup.sh`
- `.claude/actions/verify-gigabrain.sh`
- `.claude/actions/run-gigabrain-maintenance.sh`
- `.claude/actions/checkpoint-gigabrain-session.sh`

The setup is intentionally safe to rerun:

- existing `CLAUDE.md` content is preserved
- unrelated MCP servers inside `.mcp.json` are preserved
- reruns refresh only the managed Gigabrain sections

### Claude Desktop

`v0.5.3` adds two Claude Desktop extension bundle modes:

```bash
npm run claude:desktop:bundle
npm run claude:desktop:bundle:release
```

The produced `.dxt` artifact wraps the same `scripts/gigabrain-mcp.js` stdio server used by the standalone MCP path.

- `claude:desktop:bundle` is for local testing and uses the current machine's resolved standalone config path.
- `claude:desktop:bundle:release` is for GitHub/npm release assets and uses the portable default `~/.gigabrain/config.json`.

At runtime, the MCP path now expands `~`, `$HOME`, and `${HOME}` safely and falls back to an existing canonical or legacy standalone config when available.

One important pre-release hardening fix landed here too: the bundle builder now resolves and packages runtime dependencies correctly even when run from an installed npm tarball, not just from a full git checkout.

### Claude native memory vs Gigabrain

This release deliberately documents a clean boundary:

- Claude Desktop account/chat memory is Claude’s own feature and is not imported or synchronized by Gigabrain
- Claude Code still uses `CLAUDE.md` and local project instructions as its own instruction surface
- Claude Desktop Cowork is compatibility-audited only; Anthropic currently documents no memory across Cowork sessions
- Gigabrain remains the explicit, local-first memory layer for shared project/user continuity, checkpoints, provenance, and recall orchestration

## OpenClaw onboarding fix

OpenClaw onboarding now matches the current plugin loader model:

- docs now use `openclaw plugins install @legendaryvibecoder/gigabrain`
- manual instructions no longer tell users to persist `plugins.entries.gigabrain.path`
- the setup wizard now activates `plugins.slots.memory = "gigabrain"` so fresh installs actually select Gigabrain as the active memory provider
- OpenClaw docs now treat rerunning setup plus doctor as the expected recovery path for stale config

## Recall routing hardening

The recall/orchestrator work folded in from PR `#33` improves:

- identity prompts
- short preference prompts
- month-only temporal queries
- noisy metadata-wrapped prompts
- near-duplicate row selection and routing consistency

The release also fixes a contract issue discovered during the release audit: entity-brief routes backed only by world-model context now still report the correct ranking mode instead of falling back to a misleading broad mode.
